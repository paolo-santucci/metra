### Findings

#### Design canon

This is the priority section: every Flutter change must follow `Métra Screens Light.html` first, `DESIGN-BIBLE.md` second (project memory `feedback_ui_change_protocol.md`). The design files currently encode the **old** model (advance-only, no time-of-day) — the rewrite must update them in this exact order before any Dart changes.

  - line 1060: `| Preavviso | Preavviso | Notification-advance row label. **Never** "Anticipo" alone — under-specified without value. |` — keep.
  - line 1074: `| 1 giorno prima / {n} giorni prima | — | Preavviso-row value. The trailing "prima" makes the temporal relation explicit. |` — currently scoped 1–7; the rewrite extends to 1–14, no copy change needed but worth restating in the spec.

- **§ 6.7 MetraToggle** at lines 483–501 contains a load-bearing rule for the rewrite: *"If a setting requires a three-state UX, model it as a value-row with a chevron — not a toggle."* The advance row + the new time row are both correct as `value` rows; do not add a "smart" toggle that opens a picker — the chevron is the required affordance.
- **§ 18.8** "What the screen does NOT contain" (lines 1249–1253) — most consequential constraint for the planner:
  > "No bottom sheets, modals, or confirm dialogs (the mockup never renders these for any screen). **Pickers and confirmations are implementation-side concerns.**"
- **§ 15 Anti-patterns** at line 1094 reinforces it:
  > "13. **No bottom sheet** patterns in this bible's scope. (May appear in `Métra Quick Entry.html`, which is out of scope.)"

This creates a structural gap: the **current** Flutter implementation already uses a `BottomSheet` for the advance picker (see `test/features/settings/settings_screen_test.dart` line 384 `find.byType(BottomSheet)`), and the planner intends to add another picker for time-of-day. The bible currently **disowns** these as "implementation-side". For the rewrite the planner must decide:

1. **Option A (status quo)** — keep the bottom-sheet picker UX in Flutter, leave the bible silent. Add a one-line note in § 18.8 explicitly admitting that pickers are implementation-side.
2. **Option B** — promote the picker pattern to the bible as a new section (§ 19) and add the corresponding HTML component to the design system. This is the "design-first" path consistent with `feedback_ui_change_protocol.md`.

The wording "Pickers and confirmations are implementation-side concerns" is currently load-bearing to keep the canon clean, but as soon as the rewrite touches the picker (it must, for the time-of-day flow), Option B becomes the more honest path. **The planner should explicitly choose A or B at spec time so subsequent agents don't drift.**

##### Plural verification (1–14 range)

- `settings_advance_value` is already **plural-ready** in both ARB files using ICU `=1` / `other`. Italian uses `_one`/`_other` (no `_few`/`_many`), so the existing structure is correct for 1, 2, …, 14 (Italian and English both fall into `other` for n ≥ 2).
- **The planner does not need new plural categories** for the extended range — only verify in the new tests that 1, 2, 7, 8, 13, 14 all render correctly via existing plural rules.

##### Pre-existing l10n bug to fix (in scope)

`notification_prediction_body` is **not** plural — it uses a raw `{days}` placeholder. With `days = 1` the IT message renders `"La finestra stimata inizia tra 1 giorni"` (grammatically wrong, should be `"1 giorno"`); EN renders `"Your predicted window starts in 1 days"`. With the upcoming 1–14 day range and the new lower bound being still 1, this latent bug surfaces in the same code path that the rewrite touches. **Fix together with the rest** by converting both keys to `{days, plural, ...}`. Not strictly required by the new feature but cheap and adjacent — flag it in the plan, do not silently ship it.

##### Time-of-day formatting — open question

- `lib/l10n/app_en.arb` and `app_it.arb` are loaded via `gen_l10n` (`l10n.yaml` → `template-arb-file: app_it.arb`).
- A repo-wide grep for `alwaysUse24HourFormat`, `MediaQueryData`, `TimeOfDayFormat`, and `MaterialLocalizations` returns no production override (only an autogenerated comment in `lib/l10n/app_localizations.dart`). The app uses the Material default: time-of-day formatting follows the system locale + the system 24h preference.
  - IT locale defaults to 24h (`HH:mm`).
  - EN locale defaults to 12h with AM/PM (`h:mm a`) on most platforms.
- Two concrete consequences for the rewrite:
  1. The "current value" string on the new Settings row must be locale-formatted at render time. Use `MaterialLocalizations.of(context).formatTimeOfDay(time)` rather than hand-rolled `'HH:mm'`.
  2. `intl_translation` placeholder syntax `{time}` is fine if the value is already pre-formatted into a string by Dart code — do not embed time formatting inside ICU.

##### Notification-body wording risk

The current EN body is `"Your predicted window starts in {days} days"`. The IT body is `"La finestra stimata inizia tra {days} giorni"`. Neither hard-codes "tomorrow morning" / "domattina" / "9 AM" — so the new user-chosen time at, say, 23:00 does not break the wording. **No regression here**; the body is time-agnostic. Confirm during the spec phase that the planner does not introduce a "this morning" wording for the same-day immediate-show path.

### Risks

1. **No-Scrollable invariant is incompatible with 1–14 options on 360×640.** The current bottom-sheet picker (settings_screen_test lines 402, 519) explicitly forbids any `Scrollable` inside the sheet; this is a deliberate regression net (`qp-combobox-glitch` + `qp-real-device-clipping`). At 56 px row height and 14 options the picker needs ≈ 784 px before bottom-inset and dialog padding — that does not fit on a 640 px viewport. The planner must pick **one of**:
   - relax the invariant (allow scrolling), but then explicitly add a "scrollable list" UX to the bible;
   - swap the UX to a non-list picker (Material `showTimePicker`-style spinner for the day-count, or two-pane day-picker, or two-row segmented control like `1–7` / `8–14`);
   - keep 14 in a wheel (`CupertinoPicker` / `NumberPicker`) inside the sheet without a `Scrollable` widget — Cupertino wheels avoid the `Scrollable` test-finder match in some Flutter versions, but verify before committing.
   This is **the most consequential design decision of the rewrite** — surface it in the spec phase before implementation begins.
2. **DST handling for non-09 chosen times.** Italy springs forward on the **last Sunday of March** at 02:00 and falls back on the last Sunday of October. If the user picks 02:30 the spring-forward day has a non-existent local time, and the fall-back day has two valid local times. `tz.TZDateTime` resolves these per platform; the rewrite must pin the policy with explicit tests. The current single test (`notification_service_test.dart:130`) only covers 09:00, which is unambiguous.
3. **Midnight wrap.** If the user picks `00:00` and the cold-start same-day fallback compares timestamps, a notification scheduled for `00:00` on day D in a UTC-X timezone may resolve to day D-1. The existing `computeScheduledTz` test set covers 09:00 only; midnight needs its own case.
4. **CI timezone flakiness.** Tests already address this with `tz.setLocalLocation('Europe/Rome')` — keep that pattern in all new TZ-sensitive tests. Do **not** rely on `DateTime.now()` in time-of-day assertions; pass a clock override (the fake already supports `nowOverride`).
5. **Italian plural reach.** Italian uses `_one` and `_other` only — the existing `=1`/`other` ICU pattern is correct. **Risk only arises if someone adds a `_few`/`_many` form by accident** (e.g. when extending body wording). Lock this by code-review on every ARB diff.
6. **Design-canon drift.** The current Flutter advance picker uses `BottomSheet`; the bible disowns it in § 18.8 and § 15. Drift between mockup and Flutter widens with each rewrite that doesn't update the canon. The plan must update HTML mockup → bible → Flutter in that order, not the reverse. (Project memory `feedback_ui_change_protocol.md`.)
7. **Pre-existing latent bug `notification_prediction_body` (n=1)** — fixing it is in-scope and trivial, but if the rewrite ships without addressing it, IT users get `"…tra 1 giorni"` for the 1-day-advance case (the new lower bound makes `1` perfectly likely). Confirm fix is in the plan.

### Tech debt

- **`FakeNotificationService` couples to a hard-coded `9`.** Lines 60–63 encode the production policy in the fake. Once the production policy becomes parametric, the fake should track the time-of-day passed in and either receive a clock override and a chosen-time override, or compute `pastChosen = nowTime.isAfter(notifyAt)` directly off `notifyAt`. Cleaner than a duplicated boundary literal.
- **`computeScheduledTz` API is single-purpose at 09:00.** Three unit tests assert `equals(9)` literal. Promote the hour to a parameter (`computeScheduledTz(utc, hour: 9, minute: 0)`) and parametrise the tests.
- **`shouldShowImmediately` predicate hard-codes 09:00 in tests.** Same as above — once the API takes a chosen time, the four test cases each need to include a `chosenTime` so the predicate evaluates against it.
- **`notification_prediction_body` lacks ICU plural.** Pre-existing l10n bug, surfaces at `days = 1`. Fix during the rewrite.
- **Canon vs implementation tension on bottom sheets.** § 18.8 disowns pickers, but the implementation has used a `BottomSheet` for the picker since 2026-05-07 (see `qp-combobox-glitch` and `qp-real-device-clipping` checkpoints). The rewrite is the right time to either promote the pattern to the bible (Option B in the Design canon section) or formalise the disowning with a one-line note ("Pickers are implementation-side; conform to the row geometry in § 18.5 when invoked"). Either way, do not leave the gap silent.
- **`settings_notifications_on` / `_off` ARB keys' wiring not verified in this scope.** They may be dead. Verify and either delete or document why they remain.
