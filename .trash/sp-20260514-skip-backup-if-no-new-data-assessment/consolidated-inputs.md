# Consolidated Assessment — skip-backup-if-no-new-data

**Date**: 2026-05-14
**Sources**: backup-pipeline-review.md, settings-datalayer-review.md

---

## Summary

Both agents converge on the same architectural recommendation: **Option A — add a `lastLogOrSymptomWriteAt` nullable `DateTime` field to `AppSettings`**, bump it on every DailyLog/Symptom write, and compare it against `lastBackupAt` in `backupSilent()` before deciding to upload.

Options B (MAX updatedAt) and C (file mtime) are eliminated by primary evidence:
- Option B: DailyLogs/PainSymptoms have no `updatedAt` column; adding one is more invasive than A and cannot detect deletes.
- Option C: AppSettings writes on every theme/language/notification change, none of which appear in the backup blob — mtime would trigger redundant uploads on unrelated changes.

---

## Critical Findings (design constraints for the spec)

| ID | Source | Finding |
|----|--------|---------|
| C-1 | Both | No `lastLogOrSymptomWriteAt` exists; new nullable `DateTimeColumn` on `AppSettings`, schema **v8 → v9** migration (addColumn, no backfill, no downtime). |
| C-2 | Both | `_toCompanion` exclusion trap — the new field must be absent from `_toCompanion` in `drift_app_settings_repository.dart` or every `updateSettings` call silently resets it to null. |
| C-3 | Both | All 6 write methods on `DailyLogRepository` must bump the timestamp: `saveDailyLog`, `deleteDailyLog`, `replacePainSymptoms`, `deleteAll`, `deleteAllAndReplace`, `upsertAllLogs`. |
| C-4 | Both | **Restore-path loop**: `SyncOrchestrator.restore()` calls `deleteAllAndReplace()`, which would bump `lastLogOrSymptomWriteAt`. The next cold-start would see it > `lastBackupAt` and re-upload. Fix: after a successful restore, call `updateLastDataWriteAt(lastBackupAt)` so the timestamps are aligned. |

## Important Findings

| ID | Source | Finding |
|----|--------|---------|
| I-1 | Both | `lastBackupAt == null` (first-ever backup) must always upload regardless of `lastLogOrSymptomWriteAt`. |
| I-2 | Both | `lastLogOrSymptomWriteAt == null` (no data ever written) → skip; there is nothing in the DB worth uploading. |
| I-3 | Backup | Skip check belongs exclusively in `backupSilent()`; manual `backupWithPassphrase()` must always force-upload (user intent is explicit). |
| I-4 | Data | Both fakes (`FakeAppSettingsRepository`, `FakeDailyLogRepository`) must be updated in the same commit as the new field. |
| I-5 | Data | `copyWith` must NOT include `lastLogOrSymptomWriteAt` — nullable DateTime reset via copyWith is architecturally unsupported (same pattern as `lastBackupAt`, `declaredCycleLength`). |

## Suggestions

| ID | Finding |
|----|---------|
| S-1 | Name the field `lastLogOrSymptomWriteAt` (not `lastDataWriteAt`) — scope is explicitly log/symptom mutations, not settings mutations. |
| S-2 | Insertion point for the skip check: `BackupNotifier.backupSilent()` line 123, after `BackupNotConnected` guard, before passphrase read. |
| S-3 | v9 migration follows the existing `addColumn`-only pattern (no `customStatement`) from v7 and v8. |

### Patterns to follow

- `updateBackupState` / `markOnboardingComplete` / `saveDeclaredCycleLength` — dedicated writer pattern; do not reuse `updateSettings` for this field.
- `_fromRow` clamping with `.clamp()` — mirror for null-safety on read.
- v7/v8 migration blocks — addColumn only, no customStatement.

### Anti-patterns to avoid

- Do not add `lastLogOrSymptomWriteAt` to `_toCompanion`.
- Do not add it to `copyWith`.
- Do not put the skip check in `_runBackup()` — it must stay in `backupSilent()` only.

### Integration constraints

- `DriftDailyLogRepository` must hold a reference to `AppSettingsRepository` (injected via constructor or Provider) to call `updateLastDataWriteAt`. Currently it does not — this is the primary wiring change.
- Alternatively, `drift_daily_log_repository.dart` can accept a `VoidCallback` / `Future<void> Function()` to decouple the dependency direction.
- The `SyncOrchestrator` already holds `AppSettingsRepository` — the restore fix is a 1-line addition there.

### Tech debt

- `_toCompanion` exclusions are currently undocumented in code. Adding a comment when we add the new exclusion is the right moment to fix this.

### Test coverage baseline

- `backupSilent()` is currently untested directly (tested via `BackupNotifier` integration in backup_notifier_test.dart if it exists).
- `DriftDailyLogRepository` write methods are tested but without any settings-side effect verification.
- Fakes must be kept structurally consistent with real implementations or tests pass on wrong assumptions.
